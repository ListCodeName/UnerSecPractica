¿Qué es la Concurrencia?

La concurrencia es la capacidad de un sistema para gestionar múltiples tareas que progresan de forma simultánea o intercalada. No implica necesariamente que se ejecuten al mismo tiempo (eso es paralelismo), sino que el sistema puede alternar entre tareas dando la ilusión de simultaneidad.

Imaginá un cocinero que prepara dos platos: mientras uno se cocina en el horno, corta ingredientes para el otro. No está haciendo las dos cosas al mismo tiempo, pero ambas progresan. Eso es concurrencia. Si hubiera dos cocineros trabajando al mismo tiempo, eso sería paralelismo.

Hilos de Ejecución (Threads)

Un hilo (o thread) es la unidad mínima de ejecución dentro de un proceso. Un programa puede tener múltiples hilos que comparten el mismo espacio de memoria. Cada hilo ejecuta una secuencia de instrucciones de forma independiente, pero al compartir recursos con otros hilos surgen los problemas clásicos de la concurrencia.

Compartir Información

Cuando múltiples hilos necesitan acceder a los mismos datos, estamos frente a un recurso compartido. Esto es muy común en la práctica: pensá en un contador de visitas de una página web, donde cientos de solicitudes intentan leer y actualizar el mismo valor al mismo tiempo.

El problema fundamental es que las operaciones que parecen simples (como contador = contador + 1) en realidad son múltiples pasos a nivel del procesador:

// Parece una sola operación, pero internamente son 3 pasos: // 1. LEER → el hilo obtiene el valor actual (ej: 10) let temp = contador; // temp = 10 // 2. MODIFICAR → calcula el nuevo valor temp = temp + 1; // temp = 11 // 3. ESCRIBIR → guarda el resultado en memoria contador = temp; // contador = 11

Si dos hilos ejecutan estas instrucciones de forma intercalada, pueden leer el mismo valor antes de que el otro escriba, causando que una actualización se pierda. A esto se le llama condición de carrera (race condition).

Acceso de Lectura

La lectura de un recurso compartido por sí sola no genera conflictos. Múltiples hilos pueden leer el mismo dato simultáneamente sin que se corrompa la información. Pensalo así: muchas personas pueden leer el mismo cartel al mismo tiempo sin que se borre.

Lectura simultánea = segura. No hay riesgo mientras nadie modifique el recurso durante la lectura.
Acceso de Escritura

El problema aparece cuando al menos un hilo necesita escribir (modificar) el recurso compartido. Si un hilo está escribiendo mientras otro lee o escribe, los datos pueden quedar en un estado inconsistente.

Lectura + Escritura simultánea = peligrosa. El lector puede obtener un dato parcialmente actualizado. Escritura + Escritura simultánea = muy peligrosa. Ambos hilos pueden sobreescribir los cambios del otro.

Para entender la gravedad, veamos un ejemplo concreto de una condición de carrera:

Ejemplo: Race Condition con un contador (valor inicial: 10)

Hilo A: Lee: 10 → Suma: 10+1=11 → Escribe: 11
Hilo B: Lee: 10 → Suma: 10+1=11 → Escribe: 11
Resultado: Ambos leyeron 10. Ambos escribieron 11. El contador debería ser 12, pero quedó en 11. ¡Se perdió una operación!
Problemas de la Concurrencia

Los problemas de concurrencia son difíciles de detectar porque dependen del momento exacto en que cada hilo ejecuta sus instrucciones. Un programa puede funcionar correctamente 999 veces y fallar la vez 1000. Estos son los problemas clásicos:

Race Condition

Como vimos, una condición de carrera ocurre cuando el resultado de un programa depende del orden de ejecución de los hilos, un orden que no podemos controlar. El resultado se vuelve impredecible.

Deadlock (Interbloqueo)

Un deadlock ocurre cuando dos o más hilos quedan bloqueados esperando recursos que el otro tiene. Ninguno puede avanzar.

Starvation (Inanición)

La inanición sucede cuando un hilo nunca consigue acceder al recurso porque otros hilos de mayor prioridad lo acaparan constantemente. El hilo "hambriento" espera indefinidamente sin poder ejecutarse.

Datos Inconsistentes

Si las operaciones de lectura y escritura no se coordinan, los datos pueden quedar en un estado parcial o corrupto. Por ejemplo, transferir dinero entre cuentas requiere restar de una y sumar a otra: si el proceso se interrumpe a la mitad, el dinero se "pierde" o se "duplica".

JavaScript es Single-Thread

JavaScript fue diseñado en 1995 por Brendan Eich para agregar interactividad simple a las páginas web. En ese contexto, se tomó una decisión fundamental: JavaScript correrá en un único hilo de ejecución.

¿Por qué? Porque si dos hilos intentaran modificar el DOM al mismo tiempo (uno agrega un elemento, otro lo elimina), el navegador entraría en un estado caótico. Un solo hilo elimina por completo los problemas de concurrencia clásicos como race conditions y deadlocks… pero introduce otro problema: el bloqueo.

El Event Loop

JavaScript maneja las tareas asincrónicas (timers, fetch, eventos del usuario) mediante un sistema conocido como Event Loop. Este mecanismo funciona así:

Call Stack — Pila de funciones que se ejecutan de forma secuencial
Task Queue — Cola de callbacks listos para ejecutarse cuando la pila esté vacía
Event Loop — Monitorea la pila; si está vacía, toma la siguiente tarea de la cola

El Event Loop permite que JavaScript sea no-bloqueante para operaciones de I/O (leer archivos, llamadas HTTP), pero las operaciones CPU-intensivas siguen bloqueando el hilo principal.

El Problema del Bloqueo

Si ejecutamos una tarea pesada (procesar una imagen, ordenar millones de datos, calcular factoriales grandes), esa tarea congela completamente la interfaz del usuario:

Mientras el hilo principal ejecuta una tarea pesada, el navegador no puede: responder a clicks, actualizar animaciones, desplazar la página, ni renderizar cambios. El usuario ve la temida "pestaña sin respuesta".
// ❌ Esto bloquea TODA la interfaz durante segundos: function calculoPesado() { let resultado = 0; for (let i = 0; i < 2_000_000_000; i++) { resultado += Math.sqrt(i); } return resultado; // la página estuvo congelada hasta llegar acá }

Este problema fue la razón directa por la cual se crearon los Web Workers.

Web Workers

Los Web Workers (especificados en HTML5, ~2009) son la solución de JavaScript para ejecutar código en un hilo separado, sin bloquear el hilo principal del navegador.

Un Worker es esencialmente un archivo JavaScript que se ejecuta en su propio hilo. Se comunica con el hilo principal a través de un sistema de mensajes (postMessage / onmessage), sin compartir memoria directamente.

Clave: Los Workers NO comparten memoria con el hilo principal. Se comunican copiando datos mediante mensajes. Esto elimina los problemas clásicos de concurrencia (race conditions, deadlocks) por diseño.

Creación y Uso Básico
// === main.js (hilo principal) === const worker = new Worker("worker.js"); // Enviar datos al Worker worker.postMessage({ tarea: "calcular", valor: 1000000 }); // Recibir resultado del Worker worker.onmessage = function(e) { console.log("Resultado:", e.data); };
// === worker.js (hilo separado) === onmessage = function(e) { const { valor } = e.data; let suma = 0; for (let i = 0; i < valor; i++) { suma += Math.sqrt(i); } // Enviar resultado de vuelta al hilo principal postMessage(suma); };
Reglas Importantes

Los Workers tienen restricciones que previenen los problemas de concurrencia:

• No pueden acceder al DOM — no pueden modificar la página directamente.

• No comparten variables — cada Worker tiene su propia memoria aislada.

• Comunicación por mensajes — los datos se copian (no se comparten), evitando race conditions.

• Pueden usar: fetch(), setTimeout(), WebSockets, y operaciones matemáticas.
