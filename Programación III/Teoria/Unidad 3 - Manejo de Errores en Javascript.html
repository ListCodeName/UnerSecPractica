¿Qué es un Error?

Un error es una situación inesperada que interrumpe el flujo normal de un programa. Cuando JavaScript encuentra algo que no puede procesar, lanza (throws) un objeto Error que contiene información sobre qué salió mal y dónde.

Entender los errores no es solo «arreglar bugs»: es una habilidad fundamental para escribir código robusto, confiable y profesional. Un programa que no maneja errores puede fallar silenciosamente o dar resultados incorrectos sin que nadie se entere.

Anatomía de un Error en JS

Todo objeto Error en JavaScript tiene tres propiedades principales:

try { let x = variableQueNoExiste; } catch (error) { console.log(error.name); // "ReferenceError" console.log(error.message); // "variableQueNoExiste is not defined" console.log(error.stack); // traza completa del error }

📌 name: el tipo de error (ReferenceError, TypeError, etc.).

📌 message: descripción legible de qué salió mal.

📌 stack: la «traza» que muestra la línea exacta y la cadena de llamadas que produjo el error.

Tipos de Errores

Los errores se clasifican en dos grandes categorías según cuándo se detectan:

Errores de Compilación (Syntax Errors)

Se detectan antes de ejecutar el código. El motor de JavaScript analiza la sintaxis y, si encuentra algo mal escrito, rechaza el programa completo. Ninguna línea se ejecuta.

// ❌ Falta cerrar la llave function saludar() { console.log("Hola"); // } ← falta esta llave // ❌ Keyword mal escrita funtion miFuncion() { } // "funtion" en vez de "function" // ❌ Paréntesis sin cerrar console.log("Hola"; // falta el )

⚠️ Los errores de sintaxis no se pueden capturar con try/catch porque el código ni siquiera llega a ejecutarse.

Errores de Ejecución (Runtime Errors)

El código es sintácticamente correcto, pero falla durante la ejecución. Estos sí se pueden capturar y manejar con try/catch.

// ❌ Variable no definida → ReferenceError console.log(variableQueNoExiste); // ❌ Llamar algo que no es función → TypeError let x = 5; x(); // "x is not a function" // ❌ Acceder a propiedad de null → TypeError let obj = null; obj.nombre; // "Cannot read properties of null"
Errores Lógicos (los más traicioneros)

El código se ejecuta sin errores, pero produce resultados incorrectos. JavaScript no lanza ninguna excepción porque técnicamente no hay nada «roto». Estos son los más difíciles de detectar.

// ❌ Error lógico: quería sumar pero concatena function calcularTotal(precio, cantidad) { return precio + cantidad; // Si precio es "100" (string), concatena } console.log(calcularTotal("100", 5)); // "1005" en vez de 105

💡 Solución: siempre validar tipos con typeof o convertir con Number(), parseInt().

Ranking de Errores

Estos son los errores más frecuentes que encontrarás como programador JavaScript, ordenados de más común a menos frecuente:

1°

TypeError

El rey de los errores. Ocurre al operar con un tipo incorrecto: llamar a un no-función, acceder propiedades de null o undefined, usar métodos inexistentes.

2°

ReferenceError

Intentar usar una variable que no fue declarada. El clásico x is not defined. Frecuente por typos o por usar variables fuera de su scope.

3°

SyntaxError

Código mal escrito: llaves sin cerrar, comas faltantes, keywords mal escritas. El motor ni siquiera ejecuta el archivo. Frecuente pero fácil de detectar con el editor.

4°

RangeError

Un valor numérico fuera del rango permitido. Ejemplos: new Array(-1), recursión infinita que desborda la pila (Maximum call stack size exceeded).

5°

URIError

Error al usar funciones de codificación URI como decodeURIComponent('%'). Poco frecuente, aparece al manipular URLs mal formadas.

6°

EvalError

Históricamente relacionado con la función eval(). En JavaScript moderno casi no se usa y el error es muy raro. Existe principalmente por compatibilidad.

7°

InternalError (no estándar)

Error interno del motor JavaScript. El más conocido: too much recursion en Firefox. No es estándar ECMAScript, solo existe en algunos navegadores.

8°

AggregateError (ES2021)

Agrupa múltiples errores en uno solo. Se usa con Promise.any() cuando todas las promesas fallan. Es el más nuevo de la familia.

Try / Catch / Finally

La estructura try/catch/finally es el mecanismo principal para manejar errores en tiempo de ejecución en JavaScript.

Estructura básica
try { // Código que PUEDE fallar const datos = JSON.parse("esto no es JSON"); } catch (error) { // Se ejecuta SOLO si hubo error console.log("Error capturado:", error.message); } finally { // Se ejecuta SIEMPRE (haya error o no) console.log("Proceso finalizado."); }

📌 try: envuelve el código que podría fallar.

📌 catch(error): captura el error y permite manejarlo sin romper el programa.

📌 finally: se ejecuta siempre, ideal para limpieza (cerrar conexiones, ocultar loaders).

Lanzar errores con throw

Podemos crear y lanzar nuestros propios errores con throw:

function dividir(a, b) { if (b === 0) { throw new Error("No se puede dividir por cero"); } return a / b; } try { console.log(dividir(10, 0)); } catch (e) { console.log("⚠", e.message); // "No se puede dividir por cero" }
Errores Personalizados

Podemos crear clases de error propias extendiendo la clase Error. Esto permite diferenciar tipos de errores y manejar cada uno de forma específica.

Clase de Error Personalizada
class ValidacionError extends Error { constructor(campo, mensaje) { super(mensaje); this.name = "ValidacionError"; this.campo = campo; } } function registrarUsuario(nombre, edad) { if (!nombre) throw new ValidacionError("nombre", "El nombre es obligatorio"); if (edad < 0) throw new ValidacionError("edad", "La edad no puede ser negativa"); return { nombre, edad }; } try { registrarUsuario("", 25); } catch (e) { if (e instanceof ValidacionError) { console.log(`Campo: ${e.campo} → ${e.message}`); } else { throw e; // re-lanzar errores desconocidos } }
