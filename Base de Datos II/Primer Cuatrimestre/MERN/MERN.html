¿Qué es un Modelo Entidad-Relación?

La abstracción visual que da origen al diseño de bases de datos relacionales

Un Modelo Entidad-Relación (MER) es una representación gráfica que permite diseñar la estructura de una base de datos antes de implementarla. Define qué entidades existen, qué atributos tienen y cómo se relacionan entre sí.

Existen dos formas de representarlo: el diagrama ER clásico (gráfico, con figuras geométricas) y el esquema tabular compacto (moderno, más próximo a la implementación real en SQL).

🔷
ER Clásico (Chen) Rectángulos, óvalos y rombos. Cada elemento tiene su propia figura. Ideal para diseño conceptual inicial.

📋
Esquema Tabular Representación compacta en tablas. Nombre de entidad en plural, PK subrayado o marcado, FKs indicadas explícitamente.

ER Clásico — Notación Chen

Las figuras geométricas del modelo conceptual original

📌 La notación Chen (1976) usa formas geométricas específicas para representar cada elemento del modelo. Es el estándar enseñado en la mayoría de los cursos introductorios.
🔷 Elementos del diagrama
ENTIDAD

Entidad

Rectángulo. Objeto real o abstracto con existencia propia (Persona, Producto, Venta).

atributo

Atributo

Óvalo simple. Característica o propiedad de una entidad (nombre, edad, precio).

atributo

Atributo PK

Óvalo con texto subrayado. Identifica unívocamente a cada instancia (id_cliente, DNI).

multival.

Atrib. Multivaluado

Doble óvalo. Puede tener más de un valor (teléfonos, correos de una persona).

derivado

Atrib. Derivado

Óvalo punteado/discontinuo. Se calcula a partir de otro atributo (edad desde fecha_nac.).

Relación

Rombo. Vincula dos o más entidades (COMPRA, TRABAJA_EN, PERTENECE_A).

DÉBIL

Entidad Débil

Doble rectángulo. No tiene PK propia; depende de otra entidad para identificarse.

1 : N

Cardinalidad

Número junto a la línea de relación: 1:1, 1:N, N:M. Define cuántas instancias participan.

📊 Ejemplo de diagrama ER clásico — leer
Lectura de un diagrama simple: [CLIENTE] — <REALIZA> — [PEDIDO]
   |                               |
  (id_cli) (nombre) (email)    (id_ped) (fecha) (total)

Cardinalidad: 1 CLIENTE realiza N PEDIDOS → relación 1:N

Esquema Tabular Compacto

La representación moderna: más cercana al SQL real

📌 El esquema tabular reemplaza las figuras geométricas por tablas estandarizadas. El nombre de la entidad se escribe en PLURAL y en negrita en el encabezado coloreado. Cada fila representa un atributo con su PK o FK a la izquierda y el tipo de dato a la derecha.
📐 Estructura de una tabla normalizada

Estructura general:

NOMBRE_EN_PLURAL
PKid_entidadINT
atributo_1VARCHAR(100)
atributo_2DATE
FKid_otra_tablaINT

Ejemplo real — Clientes y Pedidos:

CLIENTES
PKid_clienteINT AUTO
nombreVARCHAR(80)
emailVARCHAR(120)
fecha_registroDATE
PEDIDOS
PKid_pedidoINT AUTO
FKid_clienteINT
fechaDATE
totalDECIMAL(10,2)
🔄 Correspondencia: ER Clásico → Tabla
Elemento ER Clásico Equivalente en Esquema Tabular
Rectángulo (Entidad) Tabla con encabezado en PLURAL y fondo coloreado
Óvalo simple (Atributo) Fila sin badge: nombre : VARCHAR(80)
Óvalo subrayado (PK) Fila con badge PK, generalmente la primera
Línea de relación con FK Fila con badge FK + nombre de columna referenciada
Doble óvalo (Multivaluado) Tabla aparte con PK compuesta (o nueva tabla intermedia)
Rombo N:M (Relación) Tabla intermediaria con dos FK que forman la PK compuesta
Atributo derivado (punteado) Generalmente no se incluye en la tabla; se calcula con SQL (AS, vistas)
💡 Relación N:M: Cuando dos entidades tienen relación N a M (ej: un Alumno cursa muchas Materias y una Materia tiene muchos Alumnos), se crea una tabla intermedia (ej: INSCRIPCIONES) con las FK de ambas tablas como PK compuesta.
✅

Formas Normales

Las reglas que garantizan integridad, sin redundancia ni anomalías

📌 La normalización es el proceso de organizar las tablas de una base de datos para eliminar redundancias y dependencias indeseadas. Se aplica progresivamente: cumplir la 2FN implica haber cumplido la 1FN, etc.
🔍 ¿Hasta dónde normalizar? En la práctica mayoría de sistemas bien diseñados llegan a 3FN o FNBC. Las formas 4FN y 5FN son para casos muy específicos con dependencias multivaluadas o de join, comunes en sistemas académicos o científicos.
